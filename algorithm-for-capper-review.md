# Алгоритм оценки матчей — на ревью капера

Документ описывает полный алгоритм, по которому система оценивает матчи и принимает решения о ставках. Написан для ревью: каждый шаг объяснён содержательно, не технически.

---

## Источники данных

Система работает с тремя источниками:

1. **База данных (PostgreSQL)** — статистика команд, коэффициенты букмекеров, травмы, турнирные таблицы. Данные импортируются из API-Football с задержкой до 24–48 часов.
2. **Веб-поиск (новости)** — свежие данные о составах, дисквалификациях, заявлениях тренеров. Ищутся в день матча, чтобы закрыть задержку API.
3. **Коэффициенты букмекеров** — конвертируются в подразумеваемые вероятности для выявления расхождений.

---

## Шаг 1: Отбор матчей

Берутся только матчи сегодняшнего дня со статусом `scheduled` **и** с загруженными коэффициентами. Матчи без коэффициентов или без аналитики команд пропускаются — ставка на них невозможна.

---

## Шаг 2: Загрузка статистики команд

По каждой команде из базы загружается:

### Форма (последние 20 матчей)
- Количество побед / ничьих / поражений
- Среднее забитых и пропущенных за матч

### Домашний / гостевой сплит
- **Победный процент дома** — сезонный (приоритет) и из последних 20
- **Победный процент в гостях** — аналогично

Почему сезонный приоритетнее: выборка больше, последние 20 матчей включают и домашние игры в гостевую статистику.

### Тоталы и BTTS (обе забьют)
- Процент матчей с тоталом больше 2.5 голов
- Процент матчей с тоталом больше 3.5 голов
- Процент матчей, где обе команды забили (BTTS)
- Процент матчей с сухим листом (clean sheets)

### Минуты голов
Распределение голов по периодам: 0–15, 16–30, 31–45, 46–60, 61–75, 76–90. Используется как вспомогательный контекст, не как основной сигнал.

### Статистика карточек
Распределение жёлтых и красных карточек по тем же периодам. Полезно для рынков карточек, если они доступны.

### Травмы
Игроки с активным статусом травмы — из базы (API-Football). Дополняются новостями из поиска.

### Турнирная таблица
Позиция, очки команды в лиге.

---

## Шаг 3: Поиск новостей

Для каждого матча выполняется параллельный поиск по 4 запросам:

| Запрос | Цель |
|--------|------|
| `"Команда стартовый состав дата"` | Подтверждённый состав |
| `"Команда травмы новости дата"` | Свежие травмы и дисквалификации |
| `"Команда гостей травмы дата"` | То же для гостей |
| `"Команда А vs Команда Б превью"` | Аналитика матча, цитаты тренеров |

**Что ищем в результатах:**
- Подтверждённый состав вратаря и ключевых атакующих игроков
- Дисквалификации, не отражённые в базе
- Заявления тренера о ротации / важности матча
- Усталость от дороги или плотного календаря

> Поиск новостей **обязателен** — без него алгоритм работает вслепую. Букмекер уже заложил публичную статистику в котировки; преимущество возникает на пересечении цифр со свежим контекстом.

---

## Шаг 4: Оценка вероятностей

### 4.1 Исход (1X2)

**База для хозяев:**

```
P(хозяин побеждает) = season_home_win_rate хозяина
```

Если сезонных данных нет — берётся `home_win_rate` из последних 20.

**База для гостей:**

```
P(гость побеждает) = season_away_win_rate гостя
```

**Ничья:**

```
P(ничья) = 1 - P(хозяин) - P(гость), минимум 15%
```

Минимум 15% для ничьей выставлен как страховка от статистически маловероятных, но реальных исходов.

---

**Корректировки (суммируются):**

| Условие | Сигнал | Поправка |
|---------|--------|----------|
| Средняя атака хозяев > средняя защита гостей на 0.5+ гола | Атака доминирует | +5% хозяину |
| Средняя атака гостей > средняя защита хозяев на 0.5+ | Гости атакуют лучше | +5% гостям |
| Ключевой нападающий / полузащитник хозяев травмирован | Снижение атакующей силы | −7% к голам хозяев (влияет на 1X2 и тоталы) |
| То же у гостей | — | −7% к голам гостей |
| Хозяин в топ-3, гость в нижней пятёрке | Разница мотивации | +5% хозяину |
| Гость борется за сохранение места / титул, хозяин в середине | Мотивация у гостя | +5% гостям |
| Хозяин отдыхал 4+ дня, гость играл 3 дня назад | Усталость гостя | +5% хозяину |
| Гость отдыхал 4+ дня при обратном соотношении | — | +5% гостям |
| Разница season_home_win_rate и season_away_win_rate > 20% | Явный фаворит | Удвоить вес базовой вероятности |

---

### 4.2 Тотал больше 2.5

```
P(ТБ2.5) = (total_2.5_хозяина + total_2.5_гостя) / 2
```

**Корректировки:**

| Условие | Поправка |
|---------|----------|
| Оба в среднем забивают > 1.3 гола | +5% |
| У кого-то средняя пропущенных > 1.5 (дырявая защита) | +5% |
| Оба ключевых нападающих травмированы | −10% |
| total_2.5 у обоих < 40% | Сильный сигнал ТМ2.5 |
| Дерби / финал (из новостей) | −5% (осторожный матч) |

### 4.3 Тотал больше 3.5

```
P(ТБ3.5) = (total_3.5_хозяина + total_3.5_гостя) / 2
```

Рассматривается только при коэффициенте ≥ 2.0 — иначе ожидаемая ценность слишком мала.

### 4.4 Обе забьют (BTTS)

```
P(BTTS) = (btts_хозяина + btts_гостя) / 2
```

**Корректировки:**

| Условие | Поправка |
|---------|----------|
| Хозяин держит сухой лист > 50% матчей | −8% |
| Гость держит сухой лист > 50% матчей | −8% |
| Средняя пропущенных хозяина < 0.7 | −5% |

### 4.5 Карточки (если рынок доступен)

Оцениваем по `cards_stats`:
- Высокая доля жёлтых в период 76–90 мин (отчаянные фолы в концовке)
- Средний показатель > 3 жёлтых за матч
- Контекст: матч на вылет, агрессивные команды

### 4.6 Вспомогательный контекст: минуты голов

Используется для обоснования, а не как самостоятельный сигнал:
- Команда забивает 35%+ голов в 76–90 мин → аргумент для поздних рынков
- Осторожный старт → учесть при оценке BTTS и ТБ

## Шаг 5: Поиск Value

**Подразумеваемая вероятность букмекера:**

```
P_implied = 1 / десятичный_коэффициент
```

**Очищенная от маржи вероятность** (для рынка 1X2):

```
total_implied = 1/К1 + 1/КX + 1/К2
P_fair = (1/К1) / total_implied
```

Маржа типичного букмекера — 4–8%. Очистка необходима, чтобы сравнивать наши оценки честно.

**Условие value ставки:**

```
P_наша > P_fair + 4%   →   ставка возможна (минимальный порог)
P_наша > P_fair + 8%   →   сильное value, увеличить ставку
```

Порог 4% выбран консервативно, чтобы не входить в сделки с неопределённой ценностью.

**Пропускаем ставку, если:**
- На этом типе рынка наша историческая точность < 40% (проверяется по базе)
- Нет аналитики хотя бы по одной команде
- Новости раскрыли важную информацию, которая ещё не отражена в котировках (цена сдвинется до матча — потеряем edge)

---

## Шаг 6: Размер ставки — дробный Келли 25%

```
b = десятичный_коэффициент − 1
kelly_full = (P_наша × b − (1 − P_наша)) / b
stake_pct  = kelly_full × 0.25
```

**Ограничения:**

| Ситуация | Диапазон ставки |
|----------|-----------------|
| Стандартное value (edge 4–7%) | 1.0% – 8.0% банка |
| Сильное value (edge ≥ 8%) | 2.0% – 10.0% банка |

Почему 25% от Келли: полный Келли оптимален в теории, но чувствителен к ошибкам оценки вероятностей. 25% снижает дисперсию в 4 раза при умеренном снижении доходности.

---

## Шаг 7: Управление портфелем

### На один матч
- Максимум **2 ставки** на один и тот же матч (коррелированные исходы)
- Суммарная ставка на матч — не более **12% банка**
- При превышении ставки пропорционально масштабируются вниз

### На один день
- Максимум **6 ставок**
- Максимальная экспозиция — **35% банка** в один день
- Если банк упал > 10% за последние 7 дней — все ставки уменьшаются вдвое (drawdown защита)
- Ставки ранжируются по edge; при достижении лимита убираются самые слабые

---

## Шаг 8: Формат рекомендации

По каждой ставке выводится карточка:

```
Матч, лига, время
Ставка: тип @ коэффициент

ФОРМА: W/D/L, средние голы
Сезон: % побед дома / в гостях
ТБ2.5 / BTTS %

ТАБЛИЦА: позиция и очки

ТРАВМЫ / НОВОСТИ: список травм + ключевая новость

ВЕРОЯТНОСТИ:
  Implied (без маржи): X%
  Наша оценка: Y%
  Преимущество: +Z% — value
  Стейк: N% банка = M₽
```

После всех карточек — сводная таблица дня: матч, ставка, коэф, стейк, потенциал, итого экспозиция.

**Если value ставок нет** — явно сообщаем и поясняем, почему каждый матч отклонён.

---

## Шаг 9: Запись в базу

Каждая подтверждённая ставка сохраняется в БД со статусом `pending`, со всеми деталями и текстом обоснования. Одновременно создаётся запись в журнале банкрола (`bet_placed`).

## Известные ограничения и вопросы для капера

1. **Корректировки субъективны по знаку, но не по размеру.** Поправка на травму ключевого игрока взята как −7% к голам — непонятно, откуда эта цифра. Стоит ли её откалибровать по историческим данным?

2. **Минимум ничьей 15% — жёсткий порог.** В матчах явного фаворита реальная вероятность ничьей может быть 8–10%. Правильно ли блокировать это?

3. **P(home) + P(away) могут давать сумму > 85%**, тогда ничье получает меньше 15%, но принудительно поднимается до 15% — за счёт чего пересчитываются остальные вероятности? Сейчас этот ренормализации нет.

4. **Тотал 3.5 рассматривается только при кэфе ≥ 2.0.** Логика верная, но порог условный. Можно ли его обосновать через EV?

5. **Дробный Келли 25%** — хороший стандарт для новой системы без истории. По мере накопления результатов имеет смысл перейти на 30–33%.

6. **Нет учёта формы последних 5 матчей** отдельно от последних 20. Горячая/холодная серия — важный сигнал, который сейчас теряется в общей выборке.

7. **Нет H2H (история личных встреч).** Некоторые пары команд стабильно дают конкретные паттерны (ничьи, ТМ и т.д.).

8. **Карточки и минуты голов** сейчас используются только как «вспомогательный контекст». Стоит ли формализовать условия, при которых они становятся основным сигналом?
